01这是什么

第三方开源工具，不是 TradingView 官方产品。 TradingView 没有官方 MCP，也不对外提供行情 API——所有 "TradingView MCP" 都是社区项目。它通过 Chrome DevTools Protocol 驱动你本机已登录的 TradingView 桌面应用。

ℹ️ 安装前须知
  • 第三方开源工具，不是 TradingView 官方产品。它不连服务器、不抓取、不回传任何数据，只在你本机运行。
  • 开源地址：github.com/tradesdontlie/tradingview-mcp，安装脚本已锁定成审查过的版本（55534aab）。

【步骤 5】启动 TradingView 并开启调试端口
⚠️ 这一步会关闭我当前正在运行的 TradingView，先提醒我一句再操作。

■ macOS（不要用仓库自带的 launch_tv_debug_mac.sh，也不要用 tv_launch）
  原因是【时序】，不是 pkill 或 spawn 本身有问题：这两者 kill 之后只等固定的 1 秒
  (launch_tv_debug_mac.sh:47-48) / 1.5 秒 (health.js:352-356) 就重启，旧主进程往往还没
  退干净。此时新实例通常拿不到调试端口就退出（Electron 单实例锁：新进程把参数移交旧实例
  后自行退出），9222 不会 bind，失败是静默的、退出码还是 0。
  下面流程可靠的关键在第 2 步「轮询确认进程真的退出」—— 不是 open -a 有什么魔力。
  事实上 open -a --args 在应用已运行时只会激活旧实例、丢弃 --args。

  0. 先确定 .app 路径（复用步骤 1 探测结果，不要写死 /Applications）：
       TV_APP=""
       for p in "/Applications/TradingView.app" "$HOME/Applications/TradingView.app"; do
         [ -d "$p" ] && TV_APP="$p" && break
       done
       [ -z "$TV_APP" ] && echo "未找到 TradingView.app，停止" && exit 1
     ⚠️ 必须在杀进程之前完成，否则会"应用被关了但起不来"。
  1. 找主进程 PID：ps -eo pid,args | awk '$2 ~ /Contents\/MacOS\/TradingView$/ {print $1}'
  2. kill -TERM <PID>，轮询确认真的退出（每 0.5 秒一次，最多 6 秒，没退再 kill -9）
  3. open -a "$TV_APP" --args --remote-debugging-port=9222
  4. 轮询 http://127.0.0.1:9222/json/version 直到有响应（最多 30 秒）
     open 报错就不要反复杀进程重试，把 $TV_APP 的值和完整报错发给我。

■ Windows
  ⚠️ 此时 MCP 工具尚未加载（要等重启 Claude Code 后才有），所以【不能】用 tv_launch。
     请在仓库目录下用 CLI —— 两者是同一份实现：
       cd <仓库绝对路径>
       node src/cli/index.js launch
  ⚠️ 微软商店 / MSIX 版：WindowsApps 目录 ACL 受限，直接 spawn 会 EACCES 失败；
     即使换 COM 激活，调试端口也绑不上（上游 issues #42/#75/#128）。launch 会把整个程序包
     复制到 %LOCALAPPDATA%\tradingview-mcp\<包名> 再启动，首次约 330MB、可能要一分钟。
     返回里 msix_local_copy: true 即走了这条兜底。**复制前先告诉我一声再执行。**

■ Linux（原则同 macOS：先确认旧进程完全退出，再启动新实例）
  ⚠️ 不要用 pkill -f TradingView，三个坑：
     (a) 大小写敏感 —— 匹配不到 snap 版的全小写 /snap/tradingview/current/tradingview；
     (b) 匹配【全命令行】—— 会误杀含该字样的其他进程，包括本 MCP server 自己；
     (c) Linux 的 procps pkill 只排除自身 pid、不排除祖先 —— 会 SIGTERM 掉自己的 wrapper shell。
  1. 用步骤 1 探测到的实际路径，例如 BIN=/opt/TradingView/tradingview
  2. PID=$(ps -eo pid,args | awk -v b="$BIN" '$2==b && $0 !~ /--type=/ {print $1; exit}')
     先把候选列出来给我看，确认只有 TradingView 本体。
  3. [ -n "$PID" ] && kill -TERM "$PID"，轮询 kill -0 直到消失（每 0.5 秒，最多 6 秒），
     没退再 kill -9 并等 1 秒
  4. 确认退出后：setsid "$BIN" --remote-debugging-port=9222 </dev/null >/dev/null 2>&1 &
  5. 轮询 http://127.0.0.1:9222/json/version（最多 30 秒）

■ 全平台：确认端口只绑在 127.0.0.1，不要暴露到 0.0.0.0
  macOS/Linux: lsof -nP -iTCP:9222 -sTCP:LISTEN
  Windows:     netstat -ano | findstr :9222

注：仓库路径避开 "TradingView" 字样 —— 上游 src/core/health.js:355 的 tv_launch 在非 Windows
    平台仍会执行 pkill -f TradingView；若日后调用了它，含此字样的路径会让 MCP server 自杀。

06安全加固

✓ 已由步骤 4d 自动锁死（你不用记）
规则效果
deny → tv_updateAI 调不动它。即使被工具输出里那句 "Run the tv_update tool" 忽悠，也会被硬拦
ask → ui_evaluate每次调用都必须你亲自过目，"始终允许"对它永久失效（ask 优先级高于 allow）

为什么 ui_evaluate 用 ask 而不是 deny：真正的风险不是"你批准了一次"，而是"你点了一次始终允许"——那之后提示注入随时可以得手。ask 保留了这个偶尔有用的逃生舱，同时让它永远无法进白名单。看到弹窗时要看的是：那段 JS 在干什么。尤其警惕 fetch、XMLHttpRequest、或任何外部网址。

为什么帮你禁用了自动更新（tv_update）

安装脚本已经把 tv_update 设为禁用，并把版本锁定在审查过的那一版。原因：它自动重装依赖时用的是 npm ci、没带 --ignore-scripts，万一上游改了依赖清单，可能在你电脑上跑到没审查过的第三方脚本。禁用它、用锁定版本，更稳。

要升级的时候，按下面第 7 节的手动流程走（先看改动再更新），比自动更新可控。

仍需你自己做

  • 只在使用时开 9222，用完退出 TradingView 关掉
  • 不对 pine_save、alert_*、watchlist_* 选"全部允许"（它们会改你的 TradingView 账户）
  • 不在多人共用机器上运行
  • 警惕来路不明的社区 Pine 指标（风险 2 的入口）
  • 了解 TradingView 服务条款对自动化操作有限制

端口状态检查（可选，建议装）

追加到 ~/.zshrc（bash 用户放 ~/.bashrc），开个新终端敲 tvport。它不只查端口开没开，还会校验端口是不是只绑在 127.0.0.1——万一变成 0.0.0.0（同一 WiFi 下任何人都能操控你的 TradingView），会飘红警告。

~/.zshrc
# --- TradingView MCP: 9222 调试端口状态检查 ---
tvport() {
  local line pid bind
  line=$(lsof -nP -iTCP:9222 -sTCP:LISTEN 2>/dev/null | awk 'NR==2')
  if [ -z "$line" ]; then
    printf '\033[32m○ 9222 已关闭\033[0m — TradingView 处于正常（安全）状态\n'
    return 0
  fi
  pid=$(printf '%s' "$line" | awk '{print $2}')
  bind=$(printf '%s' "$line" | awk '{print $9}')
  printf '\033[31m● 9222 调试端口开着\033[0m  PID=%s  绑定=%s\n' "$pid" "$bind"
  case "$bind" in
    127.0.0.1:*|'[::1]:'*)
      printf '  仅本机可访问（正常）。但本机任意程序仍可完全操控你已登录的 TradingView。\n' ;;
    *)
      printf '  \033[31m⚠️ 危险：未绑定 127.0.0.1，可能已暴露到网络！立即退出 TradingView。\033[0m\n' ;;
  esac
  printf '  用完请退出 TradingView（Cmd+Q），再普通双击打开即可关闭端口。\n'
}

改 .zshrc 前先备份（cp ~/.zshrc ~/.zshrc.bak），改完用 zsh -n ~/.zshrc 校验语法——语法错会导致新终端开不了。Windows 用 netstat -ano | findstr :9222。

07版本锁定与升级

本指南锁定 55534aab，因为这是被审查过的那个版本。上游后续提交未经审查。

detached HEAD 是正常的，不是故障

git checkout <完整SHA> 必然产生 detached HEAD，git 还会打印一段警告式 advice。这正是版本锁在生效的标志，不要去"修复"它。

如果跑了 tv_update，不要听它的建议。在 detached HEAD 下它会返回 "Run: git checkout main, then retry"——不要照做。若你是在上游推进之后才 clone 的，本地 main 已经是未审查的最新代码，git checkout main 当场就把你从 55534aab 扔过去。

要升级的话，手动执行，不要用 tv_update：

手动升级流程
git fetch origin main
git log --oneline 55534aab..origin/main             # 审源码提交
git diff 55534aab..origin/main -- package-lock.json # 必须单独审 lockfile
git diff 55534aab..origin/main -- package.json      # 看依赖声明有没有变
git checkout <新SHA>
npm ci --ignore-scripts --omit=dev --no-audit --no-fund
# 然后重启 Claude Code 加载新代码

审 lockfile 的红线：任何新增/变更的包，去 npmjs 上确认它没有 install 钩子；拿不准就不升级。附带好处：detached HEAD 会让 tv_update 因"不在 main 分支"直接拒绝执行（update.js:39），相当于多一道自动保险。

不信？自己复核（30 秒）

自查
cd <仓库路径>
git rev-parse HEAD                                    # 应为 55534aab8c11f24655b7d8d4de82e6bece14c8b4
grep -c '"hasInstallScript": true' package-lock.json  # 应为 0（无安装钩子）
grep -o '"resolved": "[^"]*"' package-lock.json | grep -v registry.npmjs.org   # 应无输出
⚠ 这些自查证明不了来源

比对基准来自这份文档本身，文档若被篡改则一起被篡改。真正的来源确认只能由你本人在 github.com 上手动完成。

同理，不要用 curl api.github.com/repos/.../commits/<SHA> 来"自动校验来源"：对任意自洽的 repo+commit 组合都返回 200，纯自指，只会再加一层虚假保证。

09常见问题 · 装不上看这里

装的时候，Claude 一让 TradingView 带调试端口启动就闪退 / 跳掉

TradingView 是 Electron 应用，有单实例锁。旧进程没完全退干净时带端口启动，新实例会把参数移交给旧实例、然后自己退出——就是你看到的"跳掉"。根子是旧进程没杀干净：先彻底退掉、确认真退了，再带端口启动就行。

macOS（在终端整段粘贴）：

终端 · macOS
osascript -e 'quit app "TradingView"'; sleep 3
pkill -9 -f 'TradingView.app/Contents/MacOS/TradingView' 2>/dev/null; sleep 2
lsof -iTCP:9222 -sTCP:LISTEN          # 这步没输出才正常
open -a "TradingView" --args --remote-debugging-port=9222
sleep 5; curl -s http://127.0.0.1:9222/json/version   # 看到 JSON 就成了

还跳掉，就直接跑二进制、绕过 open：

终端 · macOS 备选
"/Applications/TradingView.app/Contents/MacOS/TradingView" --remote-debugging-port=9222 >/dev/null 2>&1 &
sleep 5; curl -s http://127.0.0.1:9222/json/version

端口起来后别再普通双击打开 TradingView（会另开一个没端口的实例）。

Windows：任务管理器里彻底结束 TradingView 进程，再运行 "%LOCALAPPDATA%\TradingView\TradingView.exe" --remote-debugging-port=9222。

把链接发给 Claude，它说"做不了 / 无法在你电脑上操作"

你多半用的是 claude.ai 网页版或手机 App——它们碰不到你的电脑。必须用 Claude Code（桌面版或命令行），再把链接发给它。

它没直接装，先讲了一遍须知、还问我确认

正常的，这是特意让它先问你一声。回一句"我知道了，继续"就往下走。

它一直弹窗让我点"允许"

逐条点"允许"就行，这是安全机制，每步让你过目。别嫌烦，也别去关。
